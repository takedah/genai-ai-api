# lawsy-custom-bq 総合仕様書

## 1. 概要

### 1.1. システムの目的

本システム `lawsy-custom-bq` は、ユーザーが自然言語で投げる法令に関する質問に対し、AIが質問意図を解析し、関連する法令データを動的に検索・抽出し、レポート形式で提供するマイクロサービスである。

検索バックエンドにはGoogle Cloud BigQueryのベクトル検索機能を採用し、高いスケーラビリティとメンテナンス性を実現することを目的とする。

### 1.2. このドキュメントの対象範囲

このドキュメントは、`lawsy-custom-bq` のシステムアーキテクチャ、データ処理フロー、および開発・運用に関する技術的な仕様を定義する。開発者がシステム全体の設計思想を理解し、機能追加やメンテナンスを円滑に行えるようにすることを目的とする。

## 2. システムアーキテクチャ

### 2.1. データ3層構造

本システムは、頻繁な法令改正に対応し、データの品質と整合性を担保するため、**Source層, DWH層, App層**から成る3層のデータアーキテクチャを採用する。

```mermaid

        D --> E{5. DWH層テーブル};

        I -- 直接参照 --> H;
        J -- 検索結果 --> I;
```

*   **Source層:** e-Govから取得した生の法令XMLを構造化し、履歴としてすべて保存する層。
*   **DWH層 (Data Warehouse):** 全てのバージョンの法令データを蓄積するマスターデータ層。
*   **App層:** APIが直接参照する、最新かつクリーンなデータセットを提供する層。**新検索戦略のために2つのテーブルで構成される。**

### 2.2. 主要なGoogle Cloudサービス

*   **Cloud Functions:** `load_to_bq.py`の実行、およびAPI機能の提供。
*   **BigQuery:** データの格納、3層間のデータ変換、およびベクトル検索の実行。
*   **Cloud Storage (GCS):** `load_to_bq.py`が処理するJSONLファイルを一時的に配置するステージング領域として利用。

### 2.3. API アーキテクチャ (単一生成方式)

API（Cloud Function）のPythonコードは、単一生成方式による効率的なレポート生成を実現するため、以下のファイル構成を取ります。

```
- main.py             # エントリーポイント（単一生成ロジック実装）
- retrieval_bq.py     # BigQueryとの通信担当
- gemini_helpers.py   # Gemini API通信ヘルパー
- gemini_config.py    # AI関連設定・環境変数管理
- prompts.py          # プロンプトテンプレート定義
- schemas.py          # Pydantic データモデル定義
- genai_util.py       # レスポンス生成ユーティリティ
```

1.  **`main.py` (エントリーポイント):**
    *   Cloud FunctionsのHTTPリクエストを受け付ける起点。
    *   単一生成方式による完全なレポート生成ロジックを実装。
    *   法令名推定→並列検索→完全レポート生成の流れを統合管理。

2.  **`retrieval_bq.py` (データベース担当):**
    *   BigQueryとの通信に特化したモジュール。
    *   ベクトル検索、条文取得、法令名検索などの機能を提供。
    *   `BigQueryRetriever`クラスで統一されたDBアクセス層を提供。

3.  **`gemini_helpers.py` (AI通信担当):**
    *   Gemini APIとの通信処理を担当。
    *   リクエスト準備、API呼び出し、エラーハンドリングを統合。

4.  **`prompts.py` (プロンプト管理):**
    *   `PROMPT_GENERATE_COMPLETE_REPORT`等、単一生成用プロンプトを定義。
    *   動的構造挿入による柔軟なレポート生成を支援。

5.  **`gemini_config.py` (設定管理):**
    *   Gemini API設定、BigQuery設定等の環境変数管理。
    *   最大出力トークン数65,535等の設定を統合管理。

この構成により、従来の複雑な段階的生成から、シンプルで効率的な単一生成方式へのアーキテクチャ変更が実現されています。

## 3. データ処理フロー

### 3.1. データソース

*   **取得元:** e-Gov法令API
*   **フォーマット:** 法令標準XMLスキーマに基づいたXML形式。
*   **格納場所:** `google-cloud/lawsy-custom-bq/preprocess/all_xml/`
*   **ディレクトリ構造:** ダウンロードしたXMLは、`law_id`（例: `506M60004008009_20260822_507M60004008011`）をディレクトリ名とし、その中に同名のXMLファイル（例: `506M60004008009_20260822_507M60004008011.xml`）として格納される。

### 3.2. XMLの解析と構造化 (データ準備)

`preprocess/load_to_bq.py` スクリプトがXMLの解析と構造化を担当する。

#### 3.2.1. 法令XMLの階層構造

法令XMLは、条・項・号・号の細分といった複雑な入れ子構造を持つ。

*   **条:** `<Article>`
*   **項:** `<Paragraph>`
*   **号:** `<Item>`
*   **見出し・タイトル:** `<LawTitle>`, `<ArticleCaption>`, `<ArticleTitle>`, `<ItemTitle>`, ...
*   **文章本体:** `<ParagraphSentence>`, `<ItemSentence>`, ...

#### 3.2.2. チャンク生成ロジック

検索とLLMのコンテキスト生成のため、元の法令XMLは意味のある単位「チャンク」に分割されてBigQueryに格納される。

*   **分割単位:**
    1.  **項の柱書（はしらがき）:** `<Paragraph>`内の`<ParagraphSentence>`要素を1つのチャンクとする。
    2.  **各号:** `<Paragraph>`内の各`<Item>`要素を、それぞれ独立した1つのチャンクとする。

*   **テキスト抽出方針:**
    *   各チャンクのテキスト内容は、XMLの入れ子構造をすべて含め、文書の記述順に忠実に再現する。
    *   テキスト抽出には、XML要素が持つすべての内部テキストを順番通りに取得する `element.itertext()` を利用する。

#### 3.2.3. 追加抽出フィールド

新検索戦略のため、チャンク化の際に以下の追加フィールドがXMLから抽出され、各チャンクレコードに付与される。

*   `law_title`: 法令全体の正式名称 (`<LawTitle>`から取得)。
*   `article_caption`: チャンクが属する条の`<ArticleCaption>`（条見出し）のテキスト。存在しない場合はNULL。
*   `first_paragraph_text`: チャンクが属する条の最初の`<ParagraphSentence>`の全文テキスト。存在しない場合はNULL。

### 3.3. BigQueryへのロード (Source層)

1.  `load_to_bq.py`は、`all_xml`ディレクトリ以下の未処理のXMLファイルを特定する。
2.  各XMLファイルを上記のロジックに従って解析し、複数のJSONオブジェクト（チャンク）を生成する。
3.  全XMLファイルから生成されたJSONオブジェクトを、`data.jsonl`という単一の改行区切りJSONファイルに書き出す。
4.  `data.jsonl`をGCSバケットにアップロードする。
5.  BigQueryのロードジョブを開始し、GCS上の`data.jsonl`からSource層のテーブルにデータをロードする。
6.  **テーブル名:** `source_laws_YYYYMMDD` （実行日時点の日付サフィックスが付与される）
7.  **スキーマ (主要なもの):**
    *   `law_id` (STRING)
    *   `law_num` (STRING)
    *   `law_title` (STRING) - **(新規)**
    *   `article_caption` (STRING) - **(新規)**
    *   `first_paragraph_text` (STRING) - **(新規)**
    *   `promulgate_date` (DATE)
    *   `title` (STRING): チャンクのタイトル（例: 法令名 - 条名 - 号名）
    *   `content` (STRING): チャンクの全文テキスト
    *   `anchor` (STRING): チャンクの階層を示すアンカー

### 3.4. DWH/App層の構築

Source層のテーブルが作成された後、`preprocess/sql/`内のSQLスクリプトを実行してDWH層とApp層を更新する。

*   **DWH層 (`01_update_dwh.sql`):**
    *   **目的:** 全バージョンの法令データを履歴として蓄積する。
    *   **テーブル:** `dwh_laws`
    *   **処理:** Source層の最新テーブルから、DWH層にまだ存在しない新規・更新分のチャンクを**追記**する。

*   **App層 (`02_rebuild_app_layer.sql`):**
    *   **目的:** APIが直接参照する、最新かつクリーンな2つのデータセットを提供する。
    *   **処理:** DWH層のデータを元に、毎回テーブルを**再作成（上書き）**する。
    *   **テーブル1: `app_laws_master` (法令名検索用)**
        *   DWHからユニークな法令を抽出し、`law_title` のembedding (`law_title_embedding`) を生成して格納する。
    *   **テーブル2: `app_laws_for_indexing` (チャンク参照用)**
        *   DWHから最新版のチャンクのみを抽出する。
        *   `article_summary` カラムを `COALESCE(article_caption, SUBSTR(first_paragraph_text, 1, 40))` のロジックで動的に生成する。
